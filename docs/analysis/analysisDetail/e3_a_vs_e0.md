# E3-A 与 E0 的完整 197 题对比

## 1. 对比目的

该实验隔离“固定 train-only SQL pattern library 直接注入 Prompt”的增量效果。

```text
E0 Prompt   = 问题 + Hint + Schema + 1 个 train few-shot + DB observation
E3-A Prompt = E0 Prompt + 固定 train-only SQL pattern library
```

两组都使用：

- 固定的 `both_wrong + canary` 197 题；
- `azure/seminar-gpt-5.4-mini`、`reasoning_effort=high`、`temperature=0`；
- 1 个 train few-shot、最多 8 次 Agent 迭代；
- 相同的 DB ReAct、宽松 FINAL 协议和执行结果评测；
- 无 legacy DB hints、无 strict verified-final、无 capability gate、无递归和 Planner。

唯一计划内机制差异是 E3-A 增加 `train-static-v1` 模式库。该模式库固定写入 Prompt，不会根据当前问题检索、切片或组合，因此本实验不是 RLM 外部环境实验。

## 2. 证据边界

- E0 有两次运行，E3-A 只有一次，不能把单次差值写成稳定显著提升。
- E0 Run1 没有最终评测 SQL 超时，Run2 使用 30 秒超时，因此两次 E0 不是严格同协议复现。
- E0 Run1/Run2 分别有 9/11 次 token usage 缺失，E3-A 有 9 次；成本是已记录调用的下界。
- 语义标签来自运行后比较 predicted SQL、gold SQL 和执行答案。gold 不进入 Agent 在线决策。
- 聚合、输出列和表集合差异属于中等置信度；过滤/表达式标签置信度较低，可能包含 gold 歧义。
- 历史 E3-A manifest 没有保存精确 pattern artifact 哈希，因此与后续 E3-RF 的内容一致性只能通过当前 runner 新增的 artifact 记录约束，不能事后完全证明。

## 3. 主结果

| 运行 | 正确数 | 准确率 | 相对 E0 两次均值 |
|---|---:|---:|---:|
| E0 Run1 | 69 / 197 | 35.03% | +0.77 pp |
| E0 Run2 | 66 / 197 | 33.50% | -0.77 pp |
| E0 两次均值 | 67.5 / 197 | 34.26% | 基线 |
| E3-A | 73 / 197 | 37.06% | +2.79 pp |

E3-A 比 E0 Run1 高 4 题（+2.03 pp），比 E0 Run2 高 7 题（+3.56 pp）。这支持“固定模式库可能有小幅正增益”的探索性判断，但差值尚未经过 E3-A 重复运行验证。

前 50 题筛选中 E3-A 为 44.00%，明显高于完整 197 题的 37.06%。因此后续不能使用前 50 题结果代替完整集结论。

## 4. 分组与难度

| 子集 | E0 Run1 | E0 Run2 | E3-A | 判断 |
|---|---:|---:|---:|---|
| `both_wrong` | 19 / 137 | 17 / 137 | 22 / 137 | E3-A 分别多 3/5 题 |
| `canary` | 50 / 60 | 49 / 60 | 51 / 60 | E3-A 分别多 1/2 题 |
| simple | 26 / 50 | 26 / 50 | 27 / 50 | +1 题 |
| moderate | 27 / 96 | 27 / 96 | 31 / 96 | +4 题，主要增益来源 |
| challenging | 16 / 51 | 13 / 51 | 15 / 51 | 介于两次 E0 之间 |

E3-A 的收益主要集中在 moderate 题。challenging 题没有明确改善，因此不能声称固定模式库已经解决复杂、多阶段 SQL 推理。

## 5. 逐题配对与稳定性

| 对比 | 双方正确 | E3-A 恢复 | E3-A 回退 | 双方错误 |
|---|---:|---:|---:|---:|
| E3-A vs E0 Run1 | 65 | 8 | 4 | 120 |
| E3-A vs E0 Run2 | 63 | 10 | 3 | 121 |

相对两次 E0 都恢复的 6 题：

| ID | E0 主要原因 |
|---|---|
| `bird_1134` | 两次均为聚合错误 |
| `bird_743` | 两次均为过滤/表达式错误 |
| `bird_228` | 两次均为聚合错误 |
| `bird_27` | 两次均为过滤/表达式错误 |
| `bird_85` | 两次均为过滤/表达式错误 |
| `bird_189` | Run1 为聚合错误，Run2 为不可解析工具 observation |

相对两次 E0 都回退的只有 `bird_1171`，E3-A 将其归为过滤/表达式错误。

E0 自身有 124 道稳定失败题，其中 110 道在两次运行中保持相同语义大类，107 道保持相同子类。E3-A 只恢复了其中一小部分，说明固定模式库带来的是局部改善，没有改变大多数稳定结构性失败。

## 6. 成本与效率

| 指标 | E0 两次均值 | E3-A | 差异 |
|---|---:|---:|---:|
| LLM 调用/题 | 2.74 | 2.74 | 基本相同 |
| prompt tokens/题 | 6,461.36 | 7,394.07 | +14.4% |
| completion tokens/题 | 7,108.76 | 7,357.64 | +3.5% |
| reasoning tokens/题 | 6,553.03 | 6,785.71 | +3.6% |
| total tokens/题 | 13,570.12 | 14,751.72 | +8.7% |
| 延迟/题 | 41.80 s | 38.01 s | -9.1% |

固定模式库没有增加平均 LLM 循环次数，但增加了初始 Prompt 长度和总 token。延迟较低可能来自 API/网络波动，不能视为稳定速度优势。

E3-A 的准确率绝对提升为 2.79 pp，同时每题 total tokens 增加 8.7%。因此它当前是“可能有效但效率未改善”的候选机制，而不是已经确定的最佳 Offline 配置。

## 7. 双标签语义错误对比

首要 `error_class` 会被 `UNVERIFIED_FINAL` 覆盖，因此使用并行的 `semantic_error_class` 对比真正原因：

| 语义原因 | E0 Run1 | E0 Run2 | E0 单次范围 | E3-A | 相对 E0 的判断 |
|---|---:|---:|---:|---:|---|
| 聚合与排序 | 45 | 48 | 45-48 | 47 | 没有明确下降 |
| 过滤范围/表达式 | 33 | 34 | 33-34 | 24 | 下降 9-10 条 |
| 输出契约 | 29 | 27 | 27-29 | 29 | 没有明确下降 |
| Schema/Join | 20 | 18 | 18-20 | 20 | 没有明确下降 |
| 运行、解析或工具问题 | 1 | 4 | 1-4 | 4 | 没有改善 |
| 总失败 | 128 | 131 | 128-131 | 124 | 少 4-7 条 |

### 7.1 聚合与排序

E0 为 45/48，E3-A 为 47。模式库已经包含聚合粒度、过滤聚合、Top-K 和排序规则，但该类错误没有离开 E0 波动范围。

可能原因：

- 通用规则没有实例化为当前问题的统计对象、分组键和排序字段；
- 多阶段聚合需要显式中间计划，单条静态提醒不足；
- train few-shot 与模式规则可能重复或给出不同的输出结构；
- 模型能复述规则，但没有在 SQL 提交前进行结构化检查。

因此不能把 E3-A 的总分提升解释为聚合推理已经改善。

### 7.2 输出契约

E0 为 29/27，E3-A 为 29。虽然模式库包含输出契约规则，但输出列数和 YES/NO 形式错误没有下降。

这说明通用 Prompt 规则不足以替代题目级输出规范。后续更合适的机制是先生成结构化输出契约，例如预先记录列数、顺序、每列含义和回答形式，再生成 SQL。

### 7.3 Schema 与 Join

E0 为 20/18，E3-A 为 20。模式库没有降低选错表、遗漏表或 Join 路径错误。

这是预期风险：数据库无关的通用模式只能提醒“检查 Join”，不能告诉模型当前 Schema 中具体字段来源和外键路径。该类错误应由 metadata、Join path 检索或题目级 Schema linking 处理。

### 7.4 过滤范围与表达式

E0 为 33/34，E3-A 为 24，是唯一在总量上明显下降的语义类别。逐题恢复中，相对 Run1/Run2 分别恢复了 4/5 条该类错误，同时分别回退 1/1 条。

这支持模式库可能帮助部分边界条件、过滤层级或表达式选择。但该类别是自动分析中置信度最低的一类，也可能包含 gold 歧义；因此只能写成探索性证据，需要抽样核查恢复题的 predicted SQL 与 gold。

## 8. 控制流对比

| 控制流子类 | E0 Run1 | E0 Run2 | E3-A |
|---|---:|---:|---:|
| 错误 observation 后改写未执行 | 93 | 87 | 77 |
| observation 不可解析后改写 | 10 | 8 | 9 |
| 未调用 `db.execute` 就 FINAL | 3 | 6 | 4 |
| 正确 observation 后改写成错误 | 2 | 2 | 1 |
| 空结果后改写未执行 | 2 | 1 | 0 |
| 成功 observation 但无法对齐 | 0 | 1 | 1 |
| SQL error 后改写未执行 | 0 | 0 | 1 |
| `UNVERIFIED_FINAL` 合计 | 110 | 105 | 93 |

E3-A 的 `UNVERIFIED_FINAL` 数量下降，一部分只是因为总失败数下降。其 93 条中有 77 条的旧 SQL 已经错误，只有 1 条是正确 SQL 被改坏。因此控制流标签不能证明模式库有效，也不能推导应恢复 strict verified-final。

当前不安排独立 FINAL 同步实验。Root 仍应在正常 ReAct 中执行最终候选 SQL，但本阶段优先解决题目级语义构造问题。

## 9. 恢复与回退的原因

相对 E0 Run1：

- 恢复 8 题：过滤/表达式 4、聚合 3、Schema/Join 1；
- 回退 4 题：聚合 3、过滤/表达式 1。

相对 E0 Run2：

- 恢复 10 题：过滤/表达式 5、聚合 4、运行/工具问题 1；
- 回退 3 题：Schema/Join 1、过滤/表达式 1、输出契约 1。

聚合错误同时出现在恢复和回退中，且总体数量没有下降，说明这一部分更像逐题迁移或模型波动，而不是模式库带来的稳定类别级改善。过滤/表达式的恢复数和总体下降方向一致，是当前最值得人工复核的正向信号。

## 10. 实验判断

### 已支持的结论

1. E3-A 在单次完整运行中高于两次 E0，探索性净增益为 +2.79 pp。
2. 增益主要出现在 moderate 题。
3. E3-A 相对两次 E0 都稳定恢复 6 题、稳定回退 1 题。
4. 过滤/表达式错误数量出现下降。

### 尚未支持的结论

1. 不能声称模式库产生稳定显著提升，因为 E3-A 只有一次运行。
2. 不能声称模式库提高效率，因为 total tokens/题增加 8.7%。
3. 不能声称模式库解决聚合、输出契约或 Schema/Join；这些错误仍处于 E0 单次范围。
4. 不能声称它属于 RLM 外部环境机制；当前只是直接 Prompt 注入。
5. 不能根据 `UNVERIFIED_FINAL` 数量判断是否需要 strict verified-final 或递归。

## 11. 下一步：E3-RF

暂时保留 E3-A 作为 E3-RF 的父配置，但不将其接受为最终 Offline Query Mining 方案。

E3-RF 应保持以下变量不变：

- 同一 197 题及顺序；
- 同一模型、reasoning effort、temperature 和 max iterations；
- 同一模式库内容与 artifact 哈希；
- 同一 ReAct、数据库、评测器和 FINAL 协议；
- 只移除 train few-shot，使 effective `k=0`。

E3-RF 的判断标准：

| 结果 | 解释 |
|---|---|
| 准确率保持或提高，token 明显下降 | 模式库具有替代 train few-shot 的价值 |
| 准确率下降到 E0 范围，但 token 下降 | 模式库不能完全替代 few-shot，需要权衡效率 |
| 准确率明显低于 E0 | 固定模式库不足以独立支撑 Agent |
| 聚合/输出仍不下降 | 下一步转向题目级结构化计划和输出契约 |
| Schema/Join 上升 | 优先测试 metadata 与 Join path 检索 |
| 过滤改善在 E3-RF 中保留 | 增强模式库对过滤语义可能有独立价值 |

比较 E3-RF 时必须同时报告：

- 总准确率；
- 相对 E3-A、E0 Run1 和 E0 Run2 的 recovered/regressed IDs；
- 四类语义错误的净变化；
- 124 道 E0 稳定失败题的恢复数；
- prompt、completion、reasoning、total tokens 和调用次数；
- API、解析和不可解析 observation，避免把运行差异误认为机制收益。

## 12. 当前结论

E3-A 提供了“固定 train-only 模式可能带来小幅准确率增益”的探索性证据，但其成本更高，且没有降低聚合、输出契约或 Schema/Join 错误。当前最可信的改善信号是过滤/表达式错误减少，但该标签置信度较低。

因此，E3-A 只作为 E3-RF 的父配置。完成 E3-RF 后，再根据准确率、成本和错误类别迁移决定保留 few-shot、保留模式库，还是转向题目级结构化计划与 Schema 检索。
