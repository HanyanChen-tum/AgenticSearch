# 旧 E3-F v1/v3 与 E0、E3-A、E3-B：同题前 53 题比较

## 比较边界

E3-F run1 只完成 53/197，且只覆盖 4/11 个数据库。以下数字全部重新限制到这 53 个相同 ID，不使用完整运行的总体准确率直接比较。

E3-F 使用历史配置 `train-mined-v1 + e3-f-schema-v3 + k=1 few-shot`。它不是当前代码中的 `train-mined-v2 + e3-f-schema-v4`。

## 配对结果

| 对照 | 对照正确 | E3-F 正确 | 净变化 | E3-F 恢复 | E3-F 退化 |
|---|---:|---:|---:|---|---|
| E0 run1 | 19 | 21 | +2 | `bird_1526`, `bird_1387`, `bird_1235`, `bird_1238` | `bird_1480`, `bird_1169` |
| E0 run2 | 20 | 21 | +1 | `bird_1472`, `bird_1526`, `bird_1350` | `bird_1480`, `bird_1169` |
| E3-A | 20 | 21 | +1 | `bird_1526`, `bird_1387`, `bird_1171` | `bird_1480`, `bird_1169` |
| E3-B | 20 | 21 | +1 | `bird_1526`, `bird_1171`, `bird_1238` | `bird_1480`, `bird_1169` |

`bird_1169` 是 Azure policy `BadRequestError`，不应解释为 Offline 机制导致的语义退化。`bird_1480` 则是稳定语义退化：题目要求“peak month”，E3-F 返回完整 `YYYYMM`，而对照运行可以答对。

## 成本比较

| 实验 | 准确率 | total tokens/题 | LLM calls/题 |
|---|---:|---:|---:|
| E0 run1 | 35.85% | 11,023.26 | 2.64 |
| E0 run2 | 37.74% | 12,385.91 | 2.85 |
| E3-A | 37.74% | 11,565.96 | 2.77 |
| E3-B | 37.74% | 12,991.62 | 2.83 |
| 旧 E3-F v1/v3 | 39.62% | 16,426.79 | 3.06 |

相对 E0 两次均值，E3-F 为 +2.83 pp，但 total tokens/题为 +40.34%。相对 E3-A/E3-B 都只多答对 1 题，却分别多使用 42.03%/26.44% total tokens。

## 为什么不能把 +1 或 +2 归因给 Offline 系统

1. 只有 53 题，数据库顺序严重偏斜，且没有独立重复 run。
2. Schema v3 在这 53 题全部选择了所在数据库的所有表，不是有效的 Schema 检索消融。
3. Query Mining v1 只有 16/53 的选中 pattern 与 gold shape 完全一致，并强制 Top-K/fallback。
4. E3-F 恢复题在不同对照间不完全一致，符合模型单次运行波动；唯一共同恢复是 `bird_1526`。
5. 成本增幅远大于准确率的弱增量。

## 结论

旧 E3-F v1/v3 在前 53 题上没有形成可信的准确率或成本优势。它最有价值的结论是暴露了两个设计问题：Schema v3 退化成近似完整 Schema 注入，Query Mining v1 的检索条件过宽且没有可靠 abstain。不要继续补跑该历史配置；后续只运行独立的 E3-C Schema v4，并在 Query Mining 通过跨库门禁后再重新定义完整 E3-F。
