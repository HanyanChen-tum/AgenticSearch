# E1 verified-final 与 E0 配对比较

> 日期：2026-07-13  
> 性质：时间预算下的探索性比较  
> 比较集合：E1 落盘 71 条中的前 70 条；第 71 条不计入本报告

## 结果

| 配置 | 正确数 | 准确率 | LLM 调用/题 | Tokens/题 | 延迟/题（秒） |
|---|---:|---:|---:|---:|---:|
| E0 run1 | 29/70 | 41.43% | 2.67 | 13,255.29 | 35.18 |
| E0 run2 | 30/70 | 42.86% | 2.77 | 13,598.43 | 37.20 |
| E0 均值 | - | 42.14% | 2.72 | 13,426.86 | 36.19 |
| E1 verified-final | 28/70 | 40.00% | 5.51 | 26,920.84 | 63.23 |

E1 相对 E0 均值下降 2.14 个百分点；LLM 调用约为 2.03 倍，记录到的 token 约为 2.00 倍，延迟约为 1.75 倍。E1 有 1 次 usage 缺失，因此 token 仍是下界。

## 配对变化

- E0 两次都正确：27 题。
- E0 两次都失败：38 题。
- E0 两次结果不一致：5 题。
- E1 从稳定失败中恢复：0 题。
- E1 使稳定正确退化：3 题，分别为 `bird_1169`、`bird_1171`、`bird_1103`。
- 相对 E0 run1：恢复 2 题，退化 3 题。
- 相对 E0 run2：恢复 2 题，退化 4 题。

## 门控行为

前 70 题共有 103 次 `final.blocked`，覆盖 63 题（90%）。这说明当前门控几乎成为普遍的额外循环，而不是只处理少数未验证提交。

E0 的失败分类中，run1/run2 分别有 36/33 条 `UNVERIFIED_FINAL`；E1 中该类别归零，但失败被重新暴露为聚合、输出契约、Schema linking、runner/API 和语义错误。分类标签归零不等于准确率提升。

## 决策

拒绝当前 E1 严格 verified-final 门控：它达成了“最终 SQL 必须执行”的形式约束，但没有提高准确率，并显著增加调用、token 和延迟。后续 RLM 实验不应继承该严格门控，应回到 E0 行为作为 Root 基线。若以后重新评估最终验证，优先测试由控制器自动执行 `FINAL` SQL 的低成本版本，而不是让模型反复重新调用数据库。
